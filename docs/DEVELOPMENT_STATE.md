# AgentBridge Development State

Last updated: 2026-06-26

## Product Understanding

AgentBridge is a local-first collaboration control layer for native programming-agent CLIs. The product must preserve a real terminal session on the user's machine, expose bot-friendly semantic events, and enforce a single-writer lease so group chat users, a web admin surface, and the local human cannot corrupt the same interactive CLI state.

The authoritative design input is `AgentBridge_项目总设计文档_v0.2_多项目多会话指令版.md`.

## Important Decisions

- The first backend slice uses an in-memory repository to make command, routing, lease, and API semantics testable before introducing persistence.
- Unknown ASCII-looking `/agent` management commands are rejected instead of being silently treated as prompts. Non-command free text still becomes `ask` to support the documented shortcut pattern.
- Semantic events are separate from audit records: events drive product state replay and Bot rendering, while audit records preserve security/accountability history.
- Audit listing is a repository concern. In-memory and SQLAlchemy repositories return bounded newest-first results; the SQLAlchemy path filters action, actor, trace, project, session, interaction, created-at windows, normalized details text, and safe dot-path details-field candidates in the database, then applies the shared exact field matcher until the requested limit is reached.
- Audit archive signing now prefers an external command signer before local PEM or HMAC signing, so production deployments can bridge to KMS/HSM/Vault without loading non-exportable keys into AgentBridge. The external signer receives the canonical archive on stdin plus digest/key metadata in environment variables and returns JSON signature metadata for offline verification. `agentbridge-audit-verify` independently canonicalizes signed archives, verifies `archive_sha256`, and checks HMAC, Ed25519, RSA-PSS-SHA256, ECDSA-SHA256, or KMS-style RSA-PSS labels with a supplied HMAC key, public key, or certificate. Provider-native SDK clients and richer verifier distribution/attestation workflows remain future hardening.
- Semantic event replay and semantic event search are separate repository contracts. Replay APIs preserve per-stream ascending `seq` behavior for clients, while `/api/v1/events` and `list_semantic_events` provide bounded newest-first operational search across streams using SQLAlchemy routing, created-at, normalized payload text, database-assisted dot-path payload field candidates, and shared exact field verification. The current `q` search and field filters are useful for operations but do not replace a full JSON predicate query layer.
- SQLAlchemy persistence is currently a single-process write-through snapshot repository. Runtime connection pooling is configurable and migration deployment is documented, but multi-process production deployments still need row-level updates, stronger transaction boundaries, and explicit concurrent-writer policy.
- Terminal input must pass through the AgentBridge gateway. Direct `tmux attach` remains outside the safety model because it bypasses writer leases and Workspace write-capacity checks.
- Project `max_active_sessions` counts non-closed, non-archived, non-error sessions and is enforced before Workspace selection so over-quota requests fail without allocating short codes, sessions, or leases. Project `max_queued_turns` counts queued Turns across all sessions in the project and is enforced before Turn allocation; the returned `queue_position` shows where the rejected Turn would have landed. Project `max_running_turns` is enforced when Terminal Agent ingests `turn.started`; `turn.completed`, `turn.failed`, and `turn.interrupted` release the running slot and update the owning Session active Turn pointer. Project `daily_turns_per_user` counts all Turns queued by the same actor in the same project on the current UTC date; completed/failed/cancelled Turns still consume that daily creation quota until `reset_at`.
- Queue removal is intentionally limited to queued Turns. It marks them `cancelled` so they stop consuming queued capacity while preserving audit and event history; running Turns remain controlled by stop/interrupt lifecycle events rather than queue removal.
- Queue ordering uses a dedicated `queue_order` field rather than mutating `queued_at`, so reorder operations do not corrupt daily Turn quota accounting. Queue list responses return an opaque `queue_version` and `queue_paused`; reorder and pause/resume require the version, remove accepts it for compare-and-swap protection, clear requires both the version and a `confirm_count`/`--confirm <count>` matching the current queued Turn count, and paused queues reject new `turn.started` lifecycle events until resumed.
- `WorkspaceType.READ_ONLY` is a backend-enforced write-policy signal: creation requests are normalized to `is_writable=false` and `max_write_sessions=0` before lease enforcement.
- The local Terminal Agent socket is token-gated, chmodded to `0600`, checks same-user Unix peer credentials by default, and can reread `AGENTBRIDGE_LOCAL_TOKEN_FILE` per request for local token rotation. Windows named-pipe parity and broader production credential lifecycle policy remain future hardening work.
- Local console/daemon clients open a fresh socket per request and retry connection for short restart windows. Long offline periods still need explicit user-facing reconnect state during raw TTY passthrough.
- Console raw mode is still an input/control passthrough over the Terminal Agent socket, not a full terminal emulator. Its daemon output stream consumes backend cursor chunks; with `AGENTBRIDGE_TERMINAL_BACKEND=pty` those chunks come from a stdlib PTY reader loop, while fake/tmux remain snapshot-derived.
- The stdlib PTY backend is opt-in for local experiments. Fake remains the default test backend, and tmux remains the resumable MVP backend after Agent process restarts.
- A terminal start request without an explicit command is intentionally agent-aware: Claude Sessions launch the configured Claude command, Codex Sessions launch the configured Codex command, and generic TUI Sessions launch the configured shell command. Operators can override each profile with environment variables, lifecycle status reports whether the first executable token is currently resolvable, and terminal-control actors can explicitly run bounded version probes and adapter capability detection. Readiness checks deliberately do not execute agent commands; version probes and external handshake probes execute only on explicit POST/daemon/Admin actions with timeout and output limits. Claude/Codex structured events remain gated until the adapter handshake reports `agentbridge.adapter.v1` compatibility and a supported schema version. External adapter processes can now use the API or `agentbridge-adapter-client` to post mapped semantic events through the `agent_adapter` source only when their agent/schema match the Session, adapter approval/question/plan request events create pending Interaction records, and adapters can poll or block waiting for response frames for those interactions; protocol-native collectors and response applicators for Claude Hooks/Codex app-server still belong to the structured adapters.
- The PTY backend uses a bounded retained-output window rather than recording all terminal output. Cursor values are absolute within the session lifetime; when a reader falls behind the retained window, `read_output`/`stream_output` returns `reset=True` with the retained tail so consoles can repaint deterministically.
- `agentbridge-pty-host` is the first independent PTY host process. It keeps PTY sessions alive across API/daemon client restarts when the host process itself remains running. The host and `pty_host` client reread `AGENTBRIDGE_TERMINAL_PTY_HOST_TOKEN_FILE` per request, allowing shared token rotation without process restarts while failing closed when the token file is the only configured credential and becomes unreadable or empty. The `pty_host` client can auto-start the host and clean stale Unix sockets only when connection probing proves there is no listener; auth failures, timeouts, permission errors, and protocol errors preserve the socket and report the health error for operator review. The optional watchdog can restart a crashed host process. A host crash still destroys the owned PTY sessions, but combining the watchdog with lost-terminal auto-restart now gives an explicit restart-based recovery path from the last persisted allowlisted command. True process-preserving recovery after host death is not possible with the current stdlib PTY owner model.
- Terminal exit observation can be driven by the in-process lifecycle monitor, so clients do not have to call `status` to produce `terminal.exited`. The monitor recovers tracked terminal generations from semantic events and emits `terminal.lost` when a recovered generation has no observable backend session. `restart_session` can explicitly recover a lost/exited backend from the latest persisted command, and the opt-in auto-restart policy can do that for lost recovered terminals with a bounded attempt count after the command matches the allowlist. Service-manager deployment is now documented for the PTY host; production lifecycle supervision still needs a persistent scheduler if automatic policy execution must survive the supervising API/daemon process itself.
- The documented product-like PTY host topology is now OS service-manager first: run one `agentbridge-pty-host` per local user under systemd user services or macOS launchd, and let API/daemon clients connect over the token-gated Unix socket. In-process auto-start/watchdog remains useful as a local fallback but should not compete with an enabled service-manager unit.
- Automatic lost-terminal restart is disabled by default because restarting a native CLI can have side effects. Operators must opt in with `AGENTBRIDGE_TERMINAL_AUTO_RESTART_ON_LOST=true` and explicitly allow replayable commands with `AGENTBRIDGE_TERMINAL_AUTO_RESTART_COMMAND_ALLOWLIST`; an empty allowlist blocks automatic replay and emits `terminal.auto_restart.skipped` for operator review. `AGENTBRIDGE_TERMINAL_AUTO_RESTART_MAX_ATTEMPTS` bounds restart attempts per service process.
- Terminal lifecycle run-once is treated as an operational action because it can emit `terminal.exited`/`terminal.lost` and trigger opt-in auto-restarts. The REST endpoint requires `terminal_control` for managed device credentials and terminal-control actor permission; read-only lifecycle status still requires `audit.view` at the actor/RBAC layer and `terminal_read` for managed device credentials.
- Desktop terminal auto-open is opt-in. Custom command templates remain supported, and built-in presets cover macOS Terminal plus common Linux terminal emulators. The daemon keeps sensitive local token/socket state out of launched argv; macOS Terminal uses a mode-0700 short-lived launcher script because AppleScript cannot directly propagate the daemon's environment into the new shell.
- The tmux backend treats an existing `agentbridge_<session-id>` session as resumable state after Agent restart, matching the design's MVP recovery path.
- Rendering is split into platform-neutral documents and platform renderers. The first renderer intentionally targets text fallback so unsupported Bot platforms still receive coherent output.
- Bot delivery idempotency is implemented before real platform integration so duplicate event replay cannot cause duplicate sends once a real transport is attached.
- Bot delivery records are persisted separately from semantic events so replay, delivery retries, and platform message IDs can evolve without mutating event history.
- OneBot outbound delivery is implemented as a transport contract first. The NoneBot wrapper is optional and dependency-free; matcher registration helpers cover the common `matcher.handle()` setup path, while full NoneBot integration still needs broader lifecycle hooks, richer message components, and adapter-specific delivery capabilities.
- Delivery retry state is stored on delivery records, not events, so the immutable semantic event stream remains replayable while platform delivery can fail and recover independently.
- The retry worker reuses the Bot Gateway retry path instead of writing records directly. This keeps manual retry, background retry, and future scheduler behavior consistent.
- Platform rate-limit policies intentionally schedule unsent records as `retrying` instead of sleeping inside request handlers. This keeps API calls bounded and leaves actual waiting to the retry worker.
- Observed platform rate-limit responses are also stored as `retrying`, but keep the incremented attempt count because the platform was actually contacted.
- Interaction commands now route through the same command parser and audit chain as project/session commands. Approval voting is permission-gated by `approval.vote`; answering questions is gated by `session.send`.
- Interaction expiry is a terminal state and never auto-approves. Reads and interaction actions opportunistically advance due interactions to `expired` so pending lists do not show stale approval requests.
- Approval policy snapshots are copied onto each approval interaction so later policy changes do not rewrite historical approval requirements.
- Approval quorum overrides are intentionally scoped and snapshotted at interaction creation. Chat-context overrides win over project overrides, and explicit per-interaction `required_votes` remains the strongest override.
- OneBot inbound support currently executes text commands. The optional NoneBot wrapper can also map callback/action payloads that carry a command string through the same command execution path.
- Group role bindings are scoped to a chat context and actor ID. This keeps OneBot user permissions local to the group/private context while still allowing command/API callers to carry bootstrap roles.
- Access policy rules are stored separately from approval quorum overrides. Approval policy answers "how many votes"; access policy answers "who may do which action".
- Access policy evaluation is deny-first and then allow-before-RBAC. This makes temporary freezes explicit while preserving the existing role matrix as the default baseline.
- Access policy enforcement uses stable resource attributes only: IDs, status, project/session linkage, visibility, agent type, chat context, operation, risk level, and owner metadata. It intentionally avoids volatile or sensitive filesystem path values as policy attributes.
- Managed device identity scopes are mostly coarse transport gates (`http_api`, event WebSockets, terminal WebSocket, and Bot Gateway WebSocket), not a replacement for actor RBAC or access-policy rules. `audit_read`, `bot_gateway_read`, `bot_gateway_manage`, `onebot_event_ingest`, `command_parse`, `command_execute`, `device_manage`, `policy_read`, `policy_manage`, `group_role_read`, `group_role_manage`, `chat_context_manage`, `project_read`, `project_manage`, `session_read`, `session_manage`, `session_send`, `session_event_ingest`, `interaction_read`, `interaction_manage`, `terminal_read`, and `terminal_control` are the first action-scoped managed-device permissions; they are required for managed credentials to call audit/event history HTTP read APIs, Bot Gateway HTTP read APIs, Bot Gateway HTTP mutation APIs, OneBot inbound event APIs, command parsing, command execution, device identity APIs, access/approval policy read and simulation APIs, access/approval policy write APIs, chat-context role read APIs, chat-context role write APIs, chat-context creation/pointer write APIs, project/workspace read APIs, project/workspace/binding write APIs, session, queued-Turn, and current writer-lease read APIs, session lifecycle, writer-lease write, queue clear, queue reorder, and queue pause/resume APIs, direct session turn enqueue and queued-turn removal APIs, Terminal Agent session event ingestion APIs, interaction/approval read APIs, interaction/approval write APIs, terminal-read HTTP APIs, and terminal-control HTTP APIs. `allowed_resource_ids` can additionally bind a managed device key or managed certificate fingerprint to specific project, session, interaction, chat-context, or device-identity IDs visible in REST paths, REST query parameters, bounded top-level JSON body fields, and WebSocket route parameters; empty allowlists keep the legacy all-resource behavior, `*` is an explicit wildcard, and non-empty allowlists fail closed when a request has no visible resource ID. Command execution remains a broad entrypoint whose effects are still constrained by the command actor's RBAC and access-policy decision; direct turn enqueue and OneBot inbound commands likewise still check the effective actor permissions. Managed device identities bind device keys and active client certificate fingerprints to the same scope, resource allowlist, and revocation lifecycle, while `certificate_records` retain import/rotation/CA issuance metadata and removal timestamps for audit, renewal planning, and operations. The certificate fingerprint gates assume a trusted TLS-terminating proxy has already verified mTLS and stripped spoofable inbound fingerprint headers before setting the configured header for AgentBridge.
- The Admin Web UI is intentionally API-backed and build-free for now. It has a small `/admin` entrypoint, system health status aggregation, project/session operations for project inventory, workspace registration, session creation, session closure, active Turn/queue/pending-approval/lease status, and queue controls, interaction/approval operations for listing, filtering, creating, answering, voting, and cancelling interactions, audit/event exploration for filtered audit records plus JSON/CSV/signed-archive export and semantic event search/replay/live tail, access policy editing, terminal lifecycle inspection/run-once, device identity create/rotate/revoke, and Bot delivery operations for records, retry worker status, due retry, and rate limits. `AGENTBRIDGE_ADMIN_TOKEN(_FILE)` gates the built-in browser pages with token unlock plus an HttpOnly/SameSite cookie while preserving zero-config local development when unset. The optional REST API token gate accepts that unlocked admin cookie so browser-admin API calls are covered by the same gate, protected WebSocket event streams accept the cookie for same-origin Admin pages, and `AGENTBRIDGE_API_TOKEN(_FILE)` can serve as the admin unlock token when no separate admin token is configured. Richer dashboards remain future work.
- Session creation is evaluated against the target project because the session resource ID does not exist yet. Terminal control is evaluated as `resource_type=terminal` with the session ID so terminal-specific rules do not have to overmatch ordinary session sends.
- WebSocket session streams are read-side transports over immutable semantic events. They use `after_seq` cursors for replay/reconnect and do not mutate Bot delivery records.
- Bot Gateway WebSocket subscriptions fan out render frames for external platform adapters, but do not store delivery records. Platform adapters report delivery acknowledgements, edits, and deletes explicitly through the delivery-result API keyed by message idempotency key.
- Bot Gateway HTTP delivery can target session streams or filtered cross-stream semantic events. Cross-stream delivery requires at least one event filter so an operator cannot accidentally fan out unrelated recent events to a chat context. The certificate scan worker can automatically deliver the latest action-required scan event to configured chat contexts through the same delivery path.
- Bot delivery platform state is stored on delivery records, not semantic events. Edits/deletes update platform lifecycle metadata and latest delivery text without rewriting the immutable event stream.
- Bot Gateway edit/delete APIs call transport-native operations first, then record platform state. OneBot V11 supports deletion through `delete_msg`; message editing is a platform-specific extension and intentionally reports capability missing for the standard OneBot V11 transport.
- Render actions are emitted twice: as plain-text fallback commands and as structured button descriptors in Bot Gateway WebSocket frames. Platform adapters own conversion into native buttons/cards and should send callbacks carrying the descriptor payload command.
- `AGENTBRIDGE_API_TOKEN(_FILE)`, `AGENTBRIDGE_WS_TOKEN(_FILE)`, `AGENTBRIDGE_ADMIN_TOKEN(_FILE)`, `AGENTBRIDGE_DEVICE_KEYS`, persisted device identities, and trusted-proxy `AGENTBRIDGE_CLIENT_CERT_FINGERPRINTS(_FILE)` are the current gates for HTTP API clients, WebSocket clients, built-in admin pages, and per-device/client-certificate clients. Token and global fingerprint files are reread per request/connection and never disable a configured gate when unreadable or empty; if no other source exists, auth fails closed. Browser Admin sessions use an HttpOnly cookie for REST calls and same-origin event WebSocket streams so the unlock token does not need to be readable from JavaScript. Managed device keys are stored as salted PBKDF2 hashes, existing keys are preserved unless explicitly rotated with a non-empty `device_key`, active managed certificate fingerprints are normalized before persistence, CA-issued certificates persist serial/subject/issuer/validity metadata, renewals retire old active CA-issued fingerprints with removal metadata, add/remove rotation retains removal timestamps, and `certificate_health` reports expiring/expired/unknown certificate inventory plus renewal scheduled/due/overdue/unknown status using `AGENTBRIDGE_DEVICE_CERT_EXPIRY_WARNING_DAYS` (default 14) as the planning window. `AGENTBRIDGE_DEVICE_CERT_ISSUER_COMMAND` lets deployments delegate device certificate issuance to an external CA/KMS/HSM command; AgentBridge validates the CSR before calling it and validates the returned certificate CN, public key, expiry, clientAuth EKU, CA BasicConstraints, and direct CA signing relationship before persisting. `device_identity.certificates_scanned` is the current notification event for manual or worker-driven expiry and renewal scans, renders for operators through `GET /api/v1/events/rendered`, can be delivered to Bot chat contexts through filtered `POST /api/v1/bot-gateway/deliver-events`, and can be routed automatically by the scan worker when `AGENTBRIDGE_DEVICE_CERT_SCAN_NOTIFY_CHAT_CONTEXT_IDS` is configured; `AGENTBRIDGE_DEVICE_CERT_SCAN_WORKER_ENABLED` enables periodic scans. `docs/operations/DEVICE_CERTIFICATE_OPERATIONS.md` covers the renewal schedule/cutover runbook and current TLS proxy/CA custody boundary. Known expired managed certificate records are not accepted for fingerprint authentication; imported fingerprints without validity metadata remain allowed but are reported as unknown health. Certificate-only identities do not receive an implicit fallback key, and optional `allowed_resource_ids` constrain managed credentials to visible REST/WebSocket resource IDs including supported top-level JSON body fields bounded by `AGENTBRIDGE_DEVICE_AUTH_RESOURCE_BODY_LIMIT_BYTES` (default `1048576`). Managed device gates fail closed once any managed identity exists, even if all managed identities are revoked. These gates still need provider-native CA SDK clients, managed verifier tooling, and deeper production TLS/key lifecycle enforcement.
